# Дизайн: старт квеста и первичное распределение команд

## Цель

Добавить в бота явный старт квеста для главного администратора без станции.

После старта бот должен:

- один раз распределить команды по стартовым станциям;
- отправить первым командам сообщения для перехода на их стартовые станции;
- отправить оставшимся командам сообщение ожидания, если станций меньше, чем команд;
- скрыть кнопку старта после того, как квест уже был начат.

## Почему это нужно

Сейчас в системе есть приветственное сообщение участнику и дальнейшая маршрутизация уже в ходе квеста, но нет отдельного шага начального распределения команд по станциям.

Из-за этого боту не хватает точки, в которой:

- команды официально начинают маршрут;
- станции получают первых участников;
- `events` перестает быть пустой и система переходит из режима подготовки в режим активного квеста.

## Решение

### 1. Кнопка для главного админа

У администратора без `station_id` в меню вместо кнопки `Моя станция` появляется кнопка:

- `Начать квест`

У администратора со станцией меню не меняется:

- у него остается `Моя станция`.

Таким образом, клавиатуры `admin-home` разделяются на два вида:

- для главного админа;
- для организатора станции.

### 2. Когда кнопка доступна

Кнопка `Начать квест` должна отображаться только если:

- существует хотя бы одна команда;
- существует хотя бы одна станция;
- таблица `events` пуста.

Если `events` уже не пуста, кнопка скрывается, потому что квест считается начатым.

Если пользователь является админом станции (`station_id != NULL`), раздел для него закрыт даже при ручной отправке payload.

### 3. Подтверждение старта

При нажатии на `Начать квест` открывается модуль `assign-teams`.

Базовый текст подтверждения:

`Вы хотите распределить команды по станциям и выслать им первое сообщение?`

Кнопки:

- `Да`
- `Назад`

Если команд больше, чем станций, используется предупреждающий текст:

`Команд больше, чем активных станций. Некоторым командам придется ожидать очереди в стартовой точке. Вы точно хотите начать квест?`

Кнопки остаются те же:

- `Да`
- `Назад`

### 4. Алгоритм первичного распределения

После подтверждения:

1. Берутся все команды.
2. Берутся все станции.
3. Список станций случайно перемешивается.
4. Первые `N` команд получают первые `N` станций, где `N = min(teams, stations)`.
5. Остальные команды, если они есть, получают сообщение ожидания и переходят в `waiting_station`.

Если станций больше, чем команд:

- лишние станции просто остаются `free`;
- позже они участвуют в обычной логике выбора следующей станции.

### 5. Что означает успешный старт команды

Для команды старт считается успешным только если ей удалось доставить стартовое сообщение:

- бот отправляет `go_to_station` для назначенной станции;
- только после успешной доставки:
  - станция получает `status = occupied`;
  - в станции записывается `current_team_id`;
  - команда получает `status = on_station`;
  - у команды записывается `current_station_id`;
  - в `events` создается `active`-запись.

Это полностью совпадает с текущей моделью:

- станция считается начатой в момент назначения команде.

### 6. Если стартовое сообщение не доставлено

Если стартовое сообщение конкретной команде не доставлено:

- эта команда не считается стартовавшей;
- станция ей не назначается в БД;
- главным админам приходит лог ошибки;
- старт квеста для других команд продолжается.

Дополнительно нужна возможность повторной попытки отправить сообщения только тем командам, которые не стартовали из-за ошибки доставки.

### 7. Повторная попытка

После первичного запуска модуль `assign-teams` должен уметь повторно попытаться отправить стартовые сообщения только неуспешным командам.

Минимальная модель:

- команды, которым не удалось доставить стартовое сообщение, остаются в `waiting_start`;
- главный админ получает отдельный экран или действие повторной отправки;
- при повторной успешной доставке команда получает стартовую станцию по обычной логике:
  - выбирается свободная станция;
  - станция сразу резервируется;
  - создается `active`-событие.

В этой итерации не требуется отдельный большой модуль управления ретраями. Достаточно:

- хранить факт неуспешного старта в `user_state` главного админа или пересчитывать по данным;
- иметь явный UI-шаг, который позволяет повторить старт только для неуспешных команд.

В текущей реализации неуспешные назначения сохраняются в `user_state.payload.failedAssignments` главного админа, который запускал старт.

### 8. Что меняется в UI

#### Главный админ

До старта квеста:

- видит кнопку `Начать квест`.

После старта:

- кнопка скрывается.

#### Организатор станции

Ничего нового не получает:

- продолжает видеть `Моя станция`.

Если на станцию на старте была назначена команда:

- организатор сразу видит активную команду;
- кнопка `Завершить станцию` работает через уже существующее подтверждение.

### 9. Архитектурные изменения

#### Обновление `admin-home`

Нужно разделить клавиатуры и экраны:

- `createMainAdminKeyboard(...)`
- `createStationAdminKeyboard(...)`

Или аналогичное разделение по текущему стилю проекта.

#### Новый модуль `assign-teams`

Новый модуль должен отвечать за:

- экран подтверждения старта;
- предупреждение, если команд больше, чем станций;
- запуск случайного распределения;
- частично успешный старт;
- повторную попытку отправки неуспешным командам.

Минимальный состав:

- `src/modules/assign-teams/router.js`
- `src/modules/assign-teams/screens.js`
- `src/modules/assign-teams/keyboards.js`

#### Репозиторий событий

Текущие функции старта/перехода уже близки к нужной модели.

Может понадобиться небольшой helper для стартового назначения команды на станцию, но предпочтительно переиспользовать существующий `startStationForTeam(...)`.

### 10. Логи

Главные админы без станции получают служебные сообщения:

- какие команды успешно стартовали и на какие станции назначены;
- какие команды ушли в ожидание;
- каким командам стартовое сообщение не доставлено;
- результаты повторной попытки.

### 11. Тесты

Нужно покрыть как минимум следующие сценарии:

1. Старт квеста при `teams == stations`
   - все команды получают стартовые станции;
   - все станции становятся `occupied`;
   - `events` создаются для всех команд.

2. Старт квеста при `teams > stations`
   - первые `N` команд стартуют;
   - оставшиеся получают `wait_for_station`;
   - оставшиеся команды переходят в `waiting_station`.

3. Старт квеста при `stations > teams`
   - все команды стартуют;
   - часть станций остается `free`;
   - свободные станции дальше участвуют в обычной выборке.

4. Ошибка доставки у части стартовых команд
   - успешно доставленные команды стартуют;
   - проблемные команды не стартуют;
   - главным админам приходит лог.

5. Повторная попытка стартовой отправки
   - ранее неуспешная команда получает станцию после повторной успешной доставки;
   - создается `active`-событие;
   - станция занимаетcя сразу.

## Границы этой задачи

В эту задачу входит:

- кнопка `Начать квест`;
- двойное подтверждение;
- случайное первичное распределение;
- запуск стартовых сообщений;
- перевод лишних команд в ожидание;
- частично успешный старт;
- повторная попытка для неуспешных стартов;
- обновление тестов и документации.

В эту задачу не входит:

- ручное редактирование стартового распределения;
- визуальный предпросмотр списка стартовых назначений;
- отдельный полноценный журнал запусков квеста в БД.
