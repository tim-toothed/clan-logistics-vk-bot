# План реализации Clan Logistics VK Bot

## 1. Назначение бота

Бот нужен для проведения мероприятия со станциями и командами:

- есть список команд;
- есть список станций;
- участники получают сообщения, куда идти дальше;
- организаторы станций подтверждают завершение прохождения;
- если свободной станции нет, команда уходит в ожидание;
- если команда прошла все станции, бот отправляет финальное сообщение;
- если станцию прошли все команды, бот уведомляет организаторов этой станции;
- админы заранее настраивают команды, станции и шаблоны сообщений.

## 2. Структура проекта

- `src/main.js`
  - входная точка Worker.
- `src/app/*`
  - сбор контекста;
  - маршрутизация по `payload.action`, тексту и `state_type`.
- `src/modules/*`
  - feature-модули интерфейса:
    - `welcome`
    - `admin-home`
    - `assign-teams`
    - `my-station`
    - `setup-lists`
    - `message-templates`
    - `status`
    - `reset`
- `src/db/*`
  - репозитории D1.
- `src/utils/*`
  - VK API;
  - работа с сообщениями и вложениями;
  - доставка участникам и логирование.
- `src/ui/*`
  - общие клавиатуры и раскладка кнопок.

## 3. Минимальная схема БД

### 3.1. `users`

Хранит пользователей, открывавших бота.

Поля:

- `id`
- `vk_user_id`
- `display_name`
- `is_admin`
- `station_id`
- `team_id`

Логика:

- `is_admin = 1` — организатор или главный админ;
- `station_id IS NULL` у админа означает главного администратора без станции;
- `team_id` у участника указывает, к какой команде он относится.

### 3.2. `teams`

Поля:

- `id`
- `team_name`
- `status`
- `current_station_id`

Статусы:

- `waiting_start`
- `on_station`
- `waiting_station`
- `finished`

### 3.3. `stations`

Поля:

- `id`
- `station_name`
- `status`
- `current_team_id`

Статусы:

- `free`
- `occupied`
- `done`

### 3.4. `events`

История прохождения станций командами.

Поля:

- `id`
- `station_id`
- `team_id`
- `start_time`
- `end_time`
- `status`
- `started_by_user_id`
- `ended_by_user_id`

Статусы:

- `active`
- `completed`

Таблица используется для:

- проверки, была ли команда на станции;
- вычисления доступных станций;
- статистики по времени;
- определения, прошли ли станцию все команды;
- определения, прошла ли команда весь маршрут.

### 3.5. `messages`

Шаблоны сообщений участникам.

Поля:

- `id`
- `trigger_type`
- `station_id`
- `title`
- `content_json`
- `is_enabled`

Триггеры:

- `go_to_station`
- `wait_for_station`
- `team_finished_all`
- `participant_welcome`
- `bot_start_message`

### 3.6. `user_state`

Хранит текущее состояние диалога пользователя.

Поля:

- `id`
- `user_id`
- `state_type`
- `step_key`
- `payload_json`

## 4. Основная логика

### 4.1. Первичный старт квеста

Главный админ без станции может запустить квест кнопкой `Начать квест`.

Кнопка доступна только если:

- в системе есть хотя бы одна команда;
- в системе есть хотя бы одна станция;
- таблица `events` пуста.

При старте:

- станции случайно перемешиваются;
- в первое распределение попадают только станции, у которых `not_first = 0`;
- если станция была введена в админке как `Название*`, в базе она хранится как `Название` и получает `not_first = 1`;
- первые `N = min(teams, stations)` команд получают стартовые станции;
- остальные команды получают сообщение ожидания и переходят в `waiting_station`;
- если доставка стартового сообщения не удалась, такая команда остается в `waiting_start`;
- после частичного старта админ получает экран `Повторить отправку` только для проблемных команд.

### 4.2. Назначение станции команде

Команда получает следующую станцию в момент, когда бот отправляет сообщение `go_to_station`.

Это означает:

- станция сразу перестает быть свободной;
- у станции ставится `status = occupied`;
- у станции записывается `current_team_id`;
- у команды ставится `status = on_station`;
- у команды записывается `current_station_id`;
- в `events` создается новая запись со `status = active`.

Отдельной кнопки `Начать станцию` нет.

### 4.3. Завершение станции организатором

Организатор в разделе `Моя станция` видит назначенную команду и кнопку `Завершить станцию`.

При нажатии бот сначала показывает подтверждение:

`Подтвердите, что команда действительно прошла станцию.`

Только после подтверждения бот:

- определяет дальнейший шаг команды;
- пытается отправить участникам следующее сообщение;
- при успехе завершает текущее событие;
- либо переводит команду на новую станцию;
- либо переводит в `waiting_station`;
- либо завершает маршрут команды.

### 4.4. Защита от ошибок доставки

Если сообщение участникам не доставлено:

- станция не завершается автоматически;
- организатор остается на аварийном экране;
- появляется кнопка `Принудительно завершить`;
- главным админам без станции отправляется лог ошибки.

При `Принудительно завершить`:

- станция закрывается без успешной доставки;
- организатор получает текст для ручной передачи команде;
- главные админы получают отдельный лог.

### 4.5. Выбор следующей станции

Для команды берутся станции:

- которые она еще не проходила;
- которые сейчас `free`.

Если свободных несколько:

- выбирается станция, которая простаивает дольше всех;
- простой считается по последнему `events.end_time`.

Если свободных нет:

- команда получает сообщение ожидания;
- команде ставится `status = waiting_station`.

### 4.6. Освобождение станции и ожидание

Когда какая-то станция освобождается:

- бот проверяет команды в `waiting_station`;
- если для команды нашлась подходящая станция, бот отправляет `go_to_station`;
- после успешной доставки команда сразу переводится на эту станцию и получает `status = on_station`.

### 4.7. Завершение маршрута команды

Если команда прошла все станции:

- отправляется `team_finished_all`;
- у команды ставится `status = finished`.

### 4.8. Завершение станции для всех команд

Если станцию прошли все команды:

- у станции ставится `status = done`;
- организаторам станции отправляется сообщение.

### 4.9. Логи для главных админов

Админы без `station_id` получают служебные логи:

- итог старта квеста;
- итог повторной отправки стартовых сообщений;
- успешная доставка участникам;
- неуспешная доставка участникам;
- успешное завершение станции;
- принудительное завершение станции;
- вывод ожидающей команды на освободившуюся станцию.

Формат логов компактный:

- для успешных действий используется `🟢`;
- для ошибок и ситуаций без получателей используется `🔴`;
- для принудительного завершения сохраняется префикс `[FORCE]`;
- источник, организаторы станции и участники команды, связанные с ботом, выводятся гиперссылками на VK-диалог;
- если получателей нет, это всё равно считается проблемной ситуацией и лог помечается как `🔴`.

### 4.10. Сброс и восстановление данных

Раздел `Сброс` теперь состоит из двух отдельных сценариев:

- `История активности`
- `Все данные`

`История активности`:

- удаляет все строки из `events`;
- возвращает `teams` в `status = waiting_start` и очищает `current_station_id`;
- возвращает `stations` в `status = free` и очищает `current_team_id`;
- не удаляет команды, станции, шаблоны сообщений и пользователей;
- требует отдельного подтверждения.

`Все данные`:

- бот сначала собирает backup;
- отправляет backup-файл администраторам;
- только после успешного backup очищает `teams`, `stations`, `events`, `messages`.
- также очищает у пользователей `station_id` и `team_id`;
- требует отдельного подтверждения.

Восстановление выполняется скрытой командой `/import`.

Ограничения импорта:

- доступно только админам;
- доступно только если `teams`, `stations`, `events`, `messages` пусты;
- следующим сообщением бот ожидает файл экспорта.

## 5. Статистика

Раздел `Статистика` показывает по каждой станции:

- название станции;
- эмодзи состояния:
  - `🆓` — свободна;
  - `🎯` — занята;
  - `✅` — завершена;
- текущую команду, если станция занята;
- ссылки на участников текущей команды, которые уже зашли в бота;
- список организаторов станции в виде гиперссылок;
- сколько команд осталось;
- среднее время прохождения.

  - проверяет мягкий сброс истории активности и полный сброс всех данных.
