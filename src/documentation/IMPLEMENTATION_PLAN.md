# План реализации Clan Logistics VK Bot

## 1. Назначение бота

Бот нужен для проведения квеста:

- есть команды;
- есть станции;
- организаторы на станциях через бота отмечают начало и конец прохождения;
- после завершения станции бот сам отправляет участникам сообщение, куда идти дальше;
- если свободных станций нет, бот отправляет сообщение об ожидании;
- когда команда прошла все станции, бот отправляет финальное сообщение;
- когда станцию прошли все команды, бот отправляет сообщение организаторам этой станции;
- админы заранее заносят станции, команды и шаблоны сообщений.

Все интерфейсы и тексты бота должны быть на русском языке.

## 2. Структура проекта

- `src/main.js`
  - точка входа Worker;
  - передаёт входящие события в application layer.

- `src/app/*`
  - сбор общего контекста;
  - маршрутизация по `payload.action`, текстовым командам и `state_type`;
  - общие типы действий и состояний.

- `src/modules/*`
  - feature-модули интерфейса:
    - `welcome`
    - `admin-home`
    - `my-station`
    - `setup-lists`
    - `message-templates`
    - `status`
    - `reset`

- `src/utils/*`
  - VK API;
  - работа с сообщениями и вложениями;
  - текстовые утилиты.

- `src/ui/*`
  - общие клавиатуры и правила раскладки кнопок.

- `src/db/*`
  - работа с D1;
  - репозитории пользователей, событий, списков, сообщений и backup.

## 3. Минимальная схема БД

Для первой версии достаточно 6 таблиц.

### 3.1. `users`

Хранит пользователей, которые открыли бота.

Поля:
- `id`
- `vk_user_id`
- `display_name`
- `is_admin`
- `station_id`
- `team_id`

Логика:
- если `is_admin = 1`, это организатор/админ;
- если `is_admin = 0`, это участник;
- у админа обычно заполнен `station_id`;
- у участника обычно заполнен `team_id`;
- админ может быть без станции.

### 3.2. `teams`

Хранит команды и их текущее состояние.

Поля:
- `id`
- `team_name`
- `status`
- `current_station_id`

`status`:
- `waiting_start`
- `on_station`
- `waiting_station`
- `finished`

### 3.3. `stations`

Хранит станции и их текущее состояние.

Поля:
- `id`
- `station_name`
- `status`
- `current_team_id`

`status`:
- `free`
- `occupied`
- `done`

### 3.4. `events`

Хранит прохождение станций командами.

Поля:
- `id`
- `station_id`
- `team_id`
- `start_time`
- `end_time`
- `status`
- `started_by_user_id`
- `ended_by_user_id`

`status`:
- `active`
- `completed`

Эта таблица нужна для:
- проверки, была ли команда уже на станции;
- статистики;
- вычисления времени простоя станции;
- определения, прошли ли станцию все команды.

### 3.5. `messages`

Хранит заранее подготовленные сообщения бота для участников.

Важно:
- сообщения бывают общими;
- сообщения бывают привязанными к конкретной станции;
- одно логическое сообщение может состоять из нескольких исходящих сообщений подряд;
- внутри одного шаблона могут быть текст, фото, голосовое, аудио и другие вложения.

Поля:
- `id`
- `trigger_type`
- `station_id`
- `title`
- `content_json`
- `is_enabled`

`trigger_type`:
- `go_to_station`
- `wait_for_station`
- `team_finished_all`
- `participant_welcome`
- `bot_start_message`

Логика:

- если `trigger_type = go_to_station`, то `station_id` обязателен;
- для станции 1, 2, 3 и так далее могут быть разные сообщения;
- если сообщение общее, например ожидание, то `station_id` пустой.

Примеры:

- `go_to_station + station_id = 1`
  - сообщение для отправки на станцию 1

- `go_to_station + station_id = 2`
  - сообщение для отправки на станцию 2

- `wait_for_station + station_id = null`
  - общее сообщение об ожидании

#### Что хранится в `content_json`

В `content_json` хранится сценарий отправки.

Это массив исходящих сообщений.

Пример:

```json
[
  {
    "text": "Ваша следующая станция: Актовый зал",
    "attachments": []
  },
  {
    "text": "Ориентир по входу",
    "attachments": [
      {
        "type": "photo",
        "vk_attachment": "photo-123_456_abcd"
      }
    ]
  }
]
```

Другой пример:

```json
[
  {
    "text": "",
    "attachments": [
      {
        "type": "audio_message",
        "vk_attachment": "doc-123_789_abcd"
      }
    ]
  }
]
```

Что это позволяет:
- один текст;
- текст + картинка;
- голосовое сообщение;
- текст + аудио;
- несколько сообщений подряд.

#### Как админ будет наполнять шаблон

Админ не редактирует JSON руками.

Сценарий такой:

1. админ выбирает шаблон;
2. бот переводит его в режим записи;
3. админ присылает одно или несколько сообщений подряд;
4. бот читает текст и вложения из входящих сообщений VK;
5. бот сохраняет нормализованный результат в `content_json`.

#### Что делать с вложениями

На первом этапе не надо завязываться на R2.

Если VK уже даёт всё нужное для повторной отправки, то в `content_json` надо хранить:
- либо `vk_attachment`, пригодный для `messages.send`;
- либо нормализованный объект вложения, если это удобнее.

Идея простая:
- сначала пробуем хранить ссылки/идентификаторы VK;
- если потом окажется, что для какого-то типа вложений это неудобно или ненадёжно, только тогда добавляем R2.

То есть R2 сейчас в базовый план не включаем.

### 3.6. `user_state`

Это не лог событий.
Это память о текущем шаге диалога.

Почему нужна:
- Worker stateless;
- если админ нажал `Добавить станцию`, а потом присылает следующее сообщение с названием, боту нужно помнить, что сейчас идёт сценарий добавления станции;
- то же самое для редактирования команды, станции, шаблона сообщения или подтверждения сброса.

Поля:
- `id`
- `user_id`
- `state_type`
- `step_key`
- `payload_json`

Примеры:
- `state_type = add_station`, `step_key = wait_station_name`
- `state_type = edit_message`, `step_key = wait_message_batch`
- `state_type = reset`, `step_key = confirm`

## 4. Основная логика

### 4.1. Старт прохождения станции

Организатор нажимает `Начать прохождение`.

Бот:
- определяет станцию организатора;
- показывает только те команды, которые:
  - не находятся на другой станции;
  - ещё не проходили эту станцию.

После выбора команды:
- у станции ставится `status = occupied`;
- у станции записывается `current_team_id`;
- у команды ставится `status = on_station`;
- у команды записывается `current_station_id`;
- в `events` создаётся запись со `status = active`.

### 4.2. Завершение прохождения станции

Организатор нажимает `Завершить станцию`.

Бот:
- определяет активную команду на станции;
- завершает прохождение;
- очищает связь команды и станции;
- переводит станцию в `free` или `done`;
- запускает выбор следующей станции для команды.

После завершения:
- активная запись в `events` переводится в `completed`;
- у события заполняется `end_time`;
- у команды очищается `current_station_id`;
- у станции очищается `current_team_id`.

### 4.3. Выбор следующей станции

Для команды берутся станции:
- которые ещё не были ею пройдены;
- которые сейчас свободны.

Если свободных несколько:
- выбирается станция, которая простаивает дольше всех;
- простой считается по последнему `events.end_time`.

Если свободна одна:
- отправляем участников туда.

Если свободных нет:
- команда получает сообщение ожидания;
- у команды ставится `status = waiting_station`.

### 4.4. Завершение маршрута команды

Если команда прошла все станции:
- отправляется финальное сообщение;
- у команды ставится `status = finished`.

### 4.5. Завершение станции для всех команд

Если станцию прошли все команды:
- у станции ставится `status = done`;
- организаторам станции отправляется сообщение.

### 4.6. Сброс и восстановление данных

При сбросе мероприятия:

- бот сначала собирает backup текущих данных;
- backup отправляется администраторам как файл экспорта;
- только после успешного создания backup выполняется очистка данных.

В backup входят:

- пользователи, связанные с мероприятием;
- команды;
- станции;
- события;
- шаблоны сообщений.

Восстановление выполняется скрытой командой `/import`.

Правила импорта:

- доступно только администраторам;
- доступно только если таблицы `teams`, `stations`, `events`, `messages` пусты;
- бот ожидает следующим сообщением файл экспорта;
- поддерживаются актуальный backup-файл и legacy-JSON backup.
